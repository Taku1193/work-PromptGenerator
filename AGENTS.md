# Repository Guidelines

## プロジェクト構成

このリポジトリは、Windows 上でローカル動作する Python 製の「AI相談プロンプト生成ツール」です。利用者向けの起動入口は `app.py` で、主要ロジックは `src/` に配置します。

- `app.py`: 起動用エントリポイント
- `src/`: GUI 本体、入力フォーム、テンプレート定義、プロンプト生成処理
- `doc/`: 設計書、要件メモなどのドキュメント
- `tests/`: テストコード
- `requirements.txt`: 実行時依存関係
- `README.md`: セットアップと起動手順
- `__pycache__/`: Python 生成物。直接編集しません

フォルダー構成は以下の方針を維持してください。

- ソースコードは `src/`
- ドキュメント類は `doc/`
- テストコードは `tests/`

新規ファイル追加時も、できるだけこの構成へ寄せてください。

## ビルド・テスト・開発コマンド

- `pip install -r requirements.txt`: 依存関係をインストール
- `python app.py`: GUI を起動
- `python3 -m py_compile app.py src/*.py tests/*.py`: 構文チェック
- `python3 -m unittest discover tests`: 既存テストを実行

例:

```bash
pip install -r requirements.txt
python3 -m py_compile app.py src/*.py tests/*.py
python3 -m unittest discover tests
python app.py
```

## コーディング規約と命名

- Python の標準的な書き方に従い、インデントは 4 スペース
- クラス名は `PascalCase`
- 関数・変数は `snake_case`
- 定数は `UPPER_CASE`
- UI 表示文言や業務向けラベルは日本語を維持
- 作業種別テンプレートは `BUG_FIX_TEMPLATE` のように明示的な名前で定義し、`TEMPLATES` に登録

専用 formatter / linter は未導入です。差分は小さく保ち、既存スタイルを崩さないことを優先してください。

処理意図が読み取りにくい箇所には、簡潔でもよいので目的が分かるコメントを付けてください。特にテンプレート分岐、未入力補完、クリップボード処理は判断理由が追える状態を保ちます。

ソースコード作成時は、できるだけ詳細なコメントを残してください。特に業務ロジック、テンプレート整形、画面操作、分岐理由は、後から見ても意図が分かる粒度で補足します。

関数コメントは特に重視してください。関数定義の直前に、その関数が何のための処理か、何を返すか、どのような場面で使うかが分かるコメントをできるだけ付けてください。単なる処理の読み上げではなく、目的が伝わる書き方を優先します。

## テスト方針

現時点では `unittest` を使用します。最低限、以下を実施してください。

- コミット前に `python3 -m py_compile app.py src/*.py tests/*.py`
- `python3 -m unittest discover tests`
- GUI 変更時は `python app.py` で画面起動確認
- `プロンプト生成`、`コピー`、`クリア` の各操作を手動確認
- 未入力項目が `未入力` として出力されることを確認

テスト追加時は `tests/` 配下に `test_prompt_template.py` のような分かりやすい名前で配置してください。

## コミットとプルリクエスト

- コミットメッセージは日本語で、短く内容が分かる形で記述
  例: `プロンプト生成テンプレートを追加`
- 1コミット 1論理変更を基本とする
- PR には概要、影響範囲、手動確認内容を記載
- UI 変更がある場合はスクリーンショットを添付
- 設計変更を伴う場合は `doc/設計書.md` や `README.md` も同じ変更内で更新

## セキュリティと設定

- 外部API通信、テレメトリ、バックグラウンド通信は追加しない
- 業務上の入力内容を永続保存する実装は、要件追加がない限り行わない
- `.env` や個人用設定ファイルはコミットしない
- Windows ローカル実行を前提に、OS 依存の挙動を追加する場合は README に明記する
